# Dancelight RAG

Hybrid retrieval-augmented generation over the **2025 舞光 LED 21st 型錄**
(388-page Chinese product PDF). Combines lexical search (BM25 with a jieba
dictionary of lamp-domain terms), dense semantic embeddings, a cross-encoder
reranker, regex-extracted metadata, and a category synonym taxonomy to surface
the right product family for a free-form natural-language query.

Two entry points share the same retrieval core:

- **`dancelight_web_app.py`** — Flask web UI (port 8000). Retrieval +
  reranker only, no LLM. Returns the top-5 matching catalog chunks with the
  rendered PDF page as a thumbnail. Fast, lightweight, the path that gets
  deployed.
- **`dancelight_rag.py`** — Offline research pipeline. Full 12-step RAG with
  Ollama-served `qwen3.6` for answer generation, an annotation sidecar
  generated by `gpt-4o-mini`, and a 17-question benchmark harness against
  `Training.xlsx` / `question.xlsx`.

## Request flow (web app)

```
Browser                Flask                rag_engine                cached state
───────                ─────                ──────────                ────────────
                      [startup]
                         │
                         ├── initialize() ─────►
                         │                       load PDF + JSON
                         │                       build chunks (jieba)
                         │                       build BM25 index
                         │                       load ./bge_m3_embeddings/
                         │                              chunk_embeddings.npy ◄── 1.6 MB
                         │                       load BAAI/bge-m3 (query encoder)
                         │                       load BAAI/bge-reranker-v2-m3
                         │
                      [ready, port 8000]

POST /api/search ───►
  {query: "15W崁燈 6500K"}
                         │
                         ├── rag_engine.search(query, top_k=5)
                         │      │
                         │      ├── jieba.cut + add_synonyms  ──► tokens
                         │      ├── bm25.get_scores(tokens)   ──► bm25_scores
                         │      ├── encode(expanded_query)    ──► q_emb
                         │      ├── chunk_embeddings @ q_emb  ──► cos_scores
                         │      ├── fuse 0.5/0.5, take top 50
                         │      ├── reranker.predict(pairs)   ──► rerank top 20
                         │      └── return top 5 dicts:
                         │          {text, page, score,
                         │           page_image_url:
                         │             /api/page_image/123.png}
                         │
◄── JSON {query, results}

GET /api/page_image/123.png ──►
                         │
                         ├── PyMuPDF render page 123 @ 110 dpi
                         │   (cached after first request)
                         │
◄── image/png
```

## Files in this repo

| File | Purpose |
|------|---------|
| `dancelight_web_app.py` | Flask entry point, port 8000, 3 routes (`/`, `/api/search`, `/api/page_image/<n>.png`) |
| `rag_engine.py` | Self-contained retrieval engine: PDF → chunks → BM25 + BGE-M3 + reranker. No LLM. Used by the web app. |
| `dancelight_rag.py` | Standalone research pipeline (12 steps), uses Qwen3-Embedding-8B + bge-reranker + Ollama qwen3.6, runs benchmark and writes `rag_results.xlsx`. |
| `templates/index.html` | Single-page UI for the Flask app (Tailwind-style hand-rolled CSS). |
| `question.xlsx` / `answer.xlsx` | Benchmark queries and gold-standard model numbers (17 rows). |
| `Training.xlsx` | Larger eval set (same schema). |
| `category_synonyms.xlsx` | **Sheet 1** alias → canonical category. **Sheet 2** 大類 / 中類 / 小類 hierarchy — used to derive `SIBLINGS_MAP` for the 1.05× sibling boost. |
| `annotations_cache.json` | Pre-built per-chunk YAML sidecar (`gpt-4o-mini`, 386 entries, ~31 min + ~\$0.50 to regenerate). |
| `img_descriptions_cache.json` | Pre-built `minicpm-v` Chinese captions for catalog product images (~20 min to regenerate). |
| `bge_m3_embeddings/chunk_embeddings.npy` | 1.6 MB, 386×1024 fp32 — web app embedding cache. |
| `qwen3_8b_embeddings/chunk_embeddings.npy` | 3.1 MB, 386×4096 fp16 — research pipeline embedding cache. |
| `.env.example` | Template for the one required env var (`OPENAI_API_KEY`). |
| `requirements.txt` | Pinned Python deps. |

## Prerequisites

- **Python 3.10+**
- **The PDF** — `2025舞光LED21st(單頁水印可搜尋).pdf` placed at the repo root.
  Not in git (copyright; 168 MB). Obtain from 舞光 LED.
- **opendataloader output** — the JSON layout extraction of the same PDF,
  placed at `./output_opendataloader/2025舞光LED21st(單頁水印可搜尋).json`.
  Regenerate with [opendataloader](https://github.com/opendataloader/opendataloader).
- **Ollama** with two models pulled — only needed for the research pipeline:
  ```bash
  ollama pull minicpm-v          # vision captions (one-time, cached)
  ollama pull qwen3.6:latest     # answer generation
  ```
- **GPU recommended** for the embedding model and reranker. CPU works but is
  significantly slower; embedding the catalog from scratch on CPU takes
  >1 hour, the shipped `.npy` cache avoids this.

## Run the web app

```bash
python3 dancelight_web_app.py
# → loads PDF (~2s), builds BM25 (~1s), loads BGE-M3 + reranker (~30s on GPU)
# → serves http://0.0.0.0:8000/
```

Then in a browser open `http://<host>:8000/`, type a query
("60W 高演色性 投射燈 IP65", "15W 崁燈 6500K", etc.), and the page shows
the top-5 catalog chunks with their PDF page thumbnails.

---

## Run the research pipeline / benchmark

```bash
python3 dancelight_rag.py
# → runs all 12 steps end-to-end against question.xlsx
# → prints per-question retrieval / LLM hit rates
# → writes rag_results.xlsx (Summary + per-Q diagnostic sheets)
```

This is the full pipeline used for evaluation work — includes the annotation
sidecar, Qwen3-Embedding-8B dense retrieval, and the local qwen3.6 answer
generator. Expect ~10–15 minutes wall-clock on a single GPU.

## Configuration

Tunable constants at the top of each file.

### `rag_engine.py` (web app)

| Constant | Value | Purpose |
|----------|-------|---------|
| `EMBED_MODEL` | `BAAI/bge-m3` | Dense encoder (1024-dim). |
| `RERANK_MODEL` | `BAAI/bge-reranker-v2-m3` | CrossEncoder reranker. |
| `RETRIEVE_K` | 50 | Hybrid candidates fed to reranker. |
| `RERANK_CANDIDATES` | 20 | Top-N after rerank (web app then truncates to top-5 for UI). |
| `BM25_WEIGHT` / `VECTOR_WEIGHT` | 0.5 / 0.5 | Fusion weights. |
| `MIN_IMG_SIZE` | 30 000 bytes | Skip tiny decorative images. |

### `dancelight_rag.py` (research pipeline)

| Constant | Value | Purpose |
|----------|-------|---------|
| `EMBED_MODEL` | `Qwen/Qwen3-Embedding-8B` | Dense encoder (4096-dim, fp16). |
| `RERANK_MODEL` | `BAAI/bge-reranker-v2-m3` | Same reranker as web app. |
| `VISION_MODEL` | `minicpm-v` | Image-to-caption (Ollama). |
| `CLOUD_LLM` | `qwen3.6:latest` | Answer-generation LLM (Ollama). |
| `ANNOTATION_LLM` | `gpt-4o-mini` | Per-chunk YAML sidecar (OpenAI). |
| `TOP_K` / `RETRIEVE_K` | 20 / 50 | Final context size / hybrid candidates. |
| `BM25_WEIGHT` / `VECTOR_WEIGHT` | 0.5 / 0.5 | Fusion weights. |
| `CTX_MAX_CHARS` | 40 000 | LLM context cap (≈12 k tokens). |
| `PER_DOC_MAX` | 2 000 | Per-chunk text cap in context. |
| `NUM_PREDICT` / `NUM_CTX` | 12 000 / 32 768 | Ollama answer budget / window. |

## Caches & regeneration

Four cache artifacts ship in the repo so a fresh clone can run without
re-paying for expensive jobs. Delete the file to force a rebuild on the next
run.

| File | Size | Rebuild cost |
|------|------|--------------|
| `bge_m3_embeddings/chunk_embeddings.npy` | 1.6 MB | ~3 min GPU |
| `qwen3_8b_embeddings/chunk_embeddings.npy` | 3.1 MB | ~5 min GPU |
| `annotations_cache.json` | 324 KB | ~31 min + ~\$0.50 OpenAI |
| `img_descriptions_cache.json` | 136 KB | ~20 min on `minicpm-v` |

The PDF text and `output_opendataloader` JSON are re-parsed on every startup
(fast, <2 s combined) — they aren't cached.

## Notes

- The web app and the research pipeline use **different embedding models** by
  design — the web app prioritizes latency (BGE-M3, 1024-dim, smaller cache),
  the research pipeline prioritizes accuracy (Qwen3-Embedding-8B, 4096-dim).
  They share the same chunking, BM25, reranker, and metadata-filter code.
- The `1.05×` sibling boost is driven by Sheet 2 of `category_synonyms.xlsx`
  (the 大類 / 中類 / 小類 hierarchy). Chunks whose canonical category shares
  the same 中類 as the queried category get a small bump without being
  promoted to a full match.
- All retrieval, reranking, and answer generation paths are **offline-capable**.
  The only required cloud call is `gpt-4o-mini` for rebuilding the annotation
  sidecar — and the shipped cache makes this a one-time cost.
